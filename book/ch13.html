РОЗДІЛ 13. Більше про функції

13.1 Навіщо заглиблюватися у функції?

Ми вже ознайомилися з основами роботи функцій та розібрали поняття локальних і глобальних змінних. Однак у реальних проєктах функції рідко існують ізольовано — вони постійно взаємодіють: викликають одна одну або навіть звертаються до самих себе. Саме під час такої взаємодії керування потоком даних та видимістю змінних стає ключовим завданням для програміста.

У цьому розділі ми детально розберемо практичні сценарії:

  • чому локальна змінна, створена всередині однієї функції, є недоступною в інших частинах програми;
  • як правильно організувати виклик функції з самої себе для обчислення факторіала чи ефективного поділу складної задачі на менші підзадачі;
  • чому зміна елементів списку всередині функції може неочікувано вплинути на початковий список у головній програмі.

Вивчення матеріалу побудовано послідовно: спочатку ми дослідимо взаємодію функцій та рекурсію, після чого перейдемо до детального аналізу областей видимості змінних і поведінки складних типів даних (списків). Наприкінці розділу наведено практичні рекомендації та найкращі підходи до проектування функцій, які допоможуть уникнути типових помилок.

13.2 Виклик функції з іншої функції

При вивченні функцій ми дізнались, як їх викликати з головної програми. Насправді можна викликати функцію з іншої функції. І, більш загально, можна викликати функцію звідусіль, головне, щоб вона була видимою для Python (тобто завантаженою в пам'ять). Подивіться на цей приклад:

# Визначення функцій.
def ye_parnym(x):
    if x % 2 == 0:
        return True
    else:
        return False

def obchyslyty_sumu_parnykh(debut, kinets):
    suma = 0
    for chyslo in range(debut, kinets + 1):
        if ye_parnym(chyslo):
            suma += chyslo
    return suma

# Головна програма.
suma = obchyslyty_sumu_parnykh(1, 5)
print(f"Сума парних чисел від 1 до 5 дорівнює {suma}")

Ми викликаємо функцію obchyslyty_sumu_parnykh() з головної програми, а потім усередині неї викликаємо іншу функцію ye_parnym().

Функція може викликати іншу функцію, ця остання може викликати ще одну і так далі (і стільки разів, скільки потрібно). Функція може навіть викликати сама себе — це називається рекурсивною функцією.

У функції ye_parnym() ми перевіряємо, чи є число парним, і повертаємо True, інакше повертаємо False. Цю функцію можна написати компактніше:

def ye_parnym(x):
     return x % 2 == 0

Оскільки вираз x % 2 == 0 повертає булеве значення безпосередньо, це те саме, що й if/else вище. Звичайно, саме цю останню, компактнішу нотацію ми вам рекомендуємо.

Вправа

Створіть функцію ye_kratnym(chyslo, dilnyk), яка перевіряє, чи число кратне дільнику (за допомогою %), а також функцію pidrakhuvaty_kratni(pochatok, kinets, dilnyk), яка використовує ye_kratnym() усередині циклу, щоб порахувати, скільки чисел у діапазоні від pochatok до kinets (включно) кратні dilnyk. Перевірте функцію на діапазоні від 1 до 20 з дільником 3.

13.3 Рекурсивні функції

Рекурсивна функція — це функція, яка викликає сама себе.

Рекурсивні функції дозволяють досягти високої ефективності у розв'язанні деяких алгоритмів, таких як швидке сортування (англ. quicksort). Залишимо поки що пошук ефективності й зосередимося на прикладі математичної функції факторіал. Нагадуємо, що факторіал записується з ! і визначається так:

3! = 3 × 2 × 1 = 6
4! = 4 × 3 × 2 × 1 = 24
...
n! = n × (n-1) × ... × 2 × 1

Ось код Python з рекурсивною функцією:

def obchyslyty_faktorial(n):
    if n == 1:
        return 1
    else:
        return n * obchyslyty_faktorial(n - 1)

# Головна програма.
print(obchyslyty_faktorial(4))

У виклику print(obchyslyty_faktorial(4)) ми викликаємо функцію obchyslyty_faktorial(), передаючи як аргумент ціле число 4. Локальна змінна, яка отримує цей аргумент, — n. Усередині функція викликає сама себе, але цього разу передає значення 3. При наступному виклику це буде значення 2, а потім, нарешті, 1. В останньому випадку тест if n == 1: є істинним, і виконається інструкція return 1. У цю саму мить послідовні виклики утворюють своєрідний стек. Значення 1 буде повернено на рівень попереднього виклику, потім результат 2 × 1 = 2 (де 2 відповідає n, а 1 походить від obchyslyty_faktorial(n - 1), тобто 1) буде повернено попередньому виклику, потім 3 × 2 = 6 (де 3 відповідає n, а 2 походить від obchyslyty_faktorial(n - 1), тобто 2) буде повернено попередньому виклику, і нарешті 4 × 6 = 24 (де 4 відповідає n, а 6 походить від obchyslyty_faktorial(n - 1), тобто 6), тобто значення 4!. Послідовні виклики таким чином «розгортаються зі стека», і ми повертаємося в головну програму. Навіть якщо рекурсивні функції можуть бути складними для розуміння, наша мета тут — проілюструвати, що функція, яка викликає іншу (тут — саму себе), залишається «замороженою» в тому самому стані, доки викликана функція не поверне їй значення.

Вправа

Напишіть рекурсивну функцію obchyslyty_sumu(n), яка обчислює суму цілих чисел від 1 до n (наприклад, obchyslyty_sumu(4) має повернути 1 + 2 + 3 + 4 = 10). Порівняйте її з нерекурсивною версією, яка використовує цикл for.

13.4 Область видимості змінних

Дуже важливо при роботі з функціями знати область видимості змінних (scope англійською), тобто знати, де саме вони видимі. Ми вже бачили, що змінні, створені всередині функції, не видимі за її межами, оскільки вони локальні для функції. Подивіться на такий скрипт:

def moya_funktsiya():
    x = 2
    print(f"x дорівнює {x} у функції")

moya_funktsiya()
print(x)

🖥 Результат:

x дорівнює 2 у функції
Traceback (most recent call last):
  File "script.py", line 6, in <module>
    print(x)
NameError: name 'x' is not defined

Коли Python виконує код функції, він знає вміст змінної x. Однак, повернувшись у головну програму, він більше її не знає, звідси й повідомлення про помилку. Так само змінна, передана як аргумент, вважається локальною, щойно ми потрапляємо у функцію:

def moya_funktsiya(x):
    print(f"x дорівнює {x} у функції")

moya_funktsiya(2)
print(x)

🖥 Результат:

x дорівнює 2 у функції
Traceback (most recent call last):
  File "script.py", line 5, in <module>
    print(x)
NameError: name 'x' is not defined

Коли ж змінна оголошена в головній програмі, вона видима в ній, а також у всіх функціях. У такому разі ми говоримо про глобальну змінну:

def moya_funktsiya():
    print(x)

x = 3
moya_funktsiya()
print(x)

🖥 Результат:

3
3

У цьому випадку змінна x видима в головному модулі та в усіх функціях модуля. Однак Python не дозволяє модифікувати глобальну змінну у функції:

def moya_funktsiya():
    x = x + 1

x = 1
moya_funktsiya()

🖥 Результат:

Traceback (most recent call last):
  File "script.py", line 5, in <module>
    moya_funktsiya()
  File "script.py", line 2, in moya_funktsiya
    x = x + 1
UnboundLocalError: cannot access local variable 'x' where it is not associated with a value

Помилка, яку повертає Python, показує, що він вважає x локальною змінною, якій ще не присвоєно значення. Якщо ми справді хочемо модифікувати глобальну змінну у функції, потрібно використати ключове слово global:

def moya_funktsiya():
    global x
    x = x + 1

x = 1
moya_funktsiya()
print(x)

🖥 Результат:

2

В останньому випадку ключове слово global змусило Python вважати змінну x глобальною, а не локальною всередині функції.

Вправа

Передбачте, а потім перевірте, що виведе такий код:

def moya_funktsiya():
    y = 10
    print(y)

y = 5
moya_funktsiya()
print(y)

Потім змініть функцію так, щоб за допомогою global вона дійсно збільшувала y головної програми на 1.

13.5 Область видимості списків

Будьте надзвичайно уважні з типами, які можна модифікувати (такими як списки), оскільки їх можна змінити всередині функції:

def moya_funktsiya():
    spysok1[1] = -127

spysok1 = [1, 2, 3]
moya_funktsiya()
print(spysok1)

🖥 Результат:

[1, -127, 3]

Так само, якщо ви передаєте список як аргумент, його можна модифікувати всередині функції:

def moya_funktsiya(spysok_tmp):
    spysok_tmp[1] = -15

spysok1 = [1, 2, 3]
moya_funktsiya(spysok1)
print(spysok1)

🖥 Результат:

[1, -15, 3]

Якщо ви хочете уникнути проблем випадкової модифікації списку у функції, використовуйте кортежі (вони представлені в Розділі 8 «Словники та кортежі») — Python поверне помилку, оскільки вони незмінні. Інше рішення, щоб уникнути модифікації списку, коли він передається як аргумент функції, — це передати його явну копію (як ми робили для копіювання списку), щоб він залишився недоторканним у головній програмі:

def moya_funktsiya(spysok_tmp):
    spysok_tmp[1] = -15

spysok1 = [1, 2, 3]
moya_funktsiya(spysok1[:])
print(spysok1)

🖥 Результат:

[1, 2, 3]

У цьому прикладі копія spysok1 створюється на льоту під час виклику функції, тож список spysok1 головного модуля залишається недоторканним.

Вправа

Маючи список frukty = ["яблуко", "груша", "банан"], напишіть функцію dodaty_frukt(spysok, frukt), яка додає новий фрукт до списку за допомогою .append(). Викличте цю функцію і перевірте, що frukty дійсно змінився в головній програмі. Потім напишіть варіант функції, який замість цього повертає копію списку з доданим фруктом, не змінюючи оригінал (підказка: spysok[:]).

13.6 Правило LGI

Коли Python зустрічає змінну, він обробляє її ім'я з певними пріоритетами. Спочатку він перевіряє, чи є змінна локальною, потім, якщо її не існує локально, — чи є вона глобальною, і нарешті, якщо вона не глобальна, — чи є вона внутрішньою (наприклад, функція len() вважається внутрішньою функцією Python: вона існує щоразу, коли ви запускаєте Python). Це називається правилом LGI: локальна, глобальна, внутрішня. Ось приклад:

def moya_funktsiya():
    x = 4
    print(f"У функції x дорівнює {x}")

x = -15
moya_funktsiya()
print(f"У головному модулі x дорівнює {x}")

🖥 Результат:

У функції x дорівнює 4
У головному модулі x дорівнює -15

У функції x набув значення, яке було визначено для нього локально, з пріоритетом над значенням, визначеним у головному модулі.

Навіть якщо Python дозволяє змінній мати те саме ім'я, що й його власні функції чи внутрішні змінні, уникайте таких імен, оскільки це зробить ваш код заплутаним!

Загалом правило LGI випливає з того, як Python керує так званими «просторами імен». Саме це керування визначає область видимості (видимість) кожної змінної.

13.7 Рекомендації

13.7.1 Уникайте глобальних змінних

У цьому розділі ми гралися з функціями (та списками), щоб показати вам, як реагує Python. Однак зверніть увагу, що використання глобальних змінних варто остаточно виключити з вашої практики програмування. Іноді хочеться зробити щось швидко, і ви створюєте глобальну змінну, видиму всюди в програмі (отже, у всіх функціях), бо «так швидше, так простіше». Це дуже погана ідея, хоча б тому, що ваші функції не будуть придатні для повторного використання в іншому контексті, якщо вони спираються на глобальні змінні! А ще: чи зможете ви перечитати такий код через півроку? Чи зможе хтось інший зрозуміти вашу програму?

Замість використання глобальних змінних явно передавайте свої змінні у функції як аргумент(и).

13.7.2 Модифікація списку у функції

Щодо функцій, які модифікують список, ми радимо вам чітко вказувати це у вашому коді. Для цього зробіть так, щоб функція повертала модифікований список, і отримайте цей повернутий список у змінну з тим самим ім'ям. Наприклад:

def dodaty_odyn(spysok):
    for indeks in range(len(spysok)):
        spysok[indeks] += 1
    return spysok

# Головна програма.
spysok_otsinok = [10, 8, 16, 7, 15]
spysok_otsinok = dodaty_odyn(spysok_otsinok)
print(spysok_otsinok)

Рядок присвоєння spysok_otsinok = dodaty_odyn(spysok_otsinok) показує, що список spysok_otsinok, переданий функції, перезаписується списком, який ця функція повертає. Наступний код дасть той самий результат:

def dodaty_odyn(spysok):
    for indeks in range(len(spysok)):
        spysok[indeks] += 1

# Головна програма.
spysok_otsinok = [10, 8, 16, 7, 15]
dodaty_odyn(spysok_otsinok)
print(spysok_otsinok)

Однак це менш інтуїтивно зрозуміло, оскільки, читаючи рядок з викликом dodaty_odyn(spysok_otsinok), не очевидно, що список модифікується всередині функції. У такому разі важливо було б вказати в документації функції, що список модифікується «на місці» (in place англійською).

З наведених вище причин ми радимо надавати перевагу першому варіанту:

spysok_otsinok = dodaty_odyn(spysok_otsinok)
